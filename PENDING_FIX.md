# 待修:`json_string` 逗號 bug(閘門回來後立即處理)

> 由 wpdlypl7h 的安全軌明確標示「out of scope, not fixed」。
> 不在此刻修的原因:第三輪閘門正在稽核當前檔案,改 `install.sh` 會讓稽核對象變動,
> 且會使四個 `.sha256` 再次失效(那是 critical 級項目)。

## 現況

`site/install.sh:709-713`

```sh
json_string() {
    tr ',' '\n' | \
        sed -n 's/.*"'"$1"'"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | \
        head -n 1
}
```

`tr ',' '\n'` 在抽取**之前**就把 JSON 依逗號切開。任何含逗號的字串值會被切成兩半,
剩下的片段沒有結尾引號,sed 匹配不到 → 靜默回空。

實例:`{"message":"install complete, key accepted"}` → 訊息整段消失。

## 影響

- 伺服器回的 `message` / `type` 只要含逗號就**完全不顯示**,使用者看不到失敗原因。
- 這直接抵銷 HIGH-3 那組消毒工作 —— 消毒的前提是訊息會被顯示出來。
- 失敗方向是安全的(回空而非亂碼),所以是可用性缺陷,不是資安漏洞。

## 修法

不要切,直接抓最短匹配。`[^"]*` 本來就不會跨越引號,所以值裡的逗號完全無害:

```sh
json_string() {
    # 不要先 tr ',' '\n' —— 那會把含逗號的值切斷,讓它靜默消失。
    # grep -o 取最短匹配;[^"]* 不跨引號,所以值內的逗號無害。
    grep -o "\"$1\"[[:space:]]*:[[:space:]]*\"[^\"]*\"" | \
        head -n 1 | \
        sed 's/.*:[[:space:]]*"//; s/"$//'
}
```

已知限制(沿用原本行為,失敗方向安全):值內含跳脫引號 `\"` 時會截斷。

`install.ps1` 用的是 .NET 的 JSON 解析,不受此影響 —— 需確認後再決定是否同步調整。

## 修完必做

1. `sh -n` / `bash -n` / `dash -n`
2. 加一個測試:含逗號的 message 要能正確取出
3. **重算四個 `.sha256`**,並確認磁碟 = git blob = 發佈摘要三方一致(純 LF)
4. 注入回歸重跑
5. 重新過閘門(改了 `install.sh` 就必須重驗)

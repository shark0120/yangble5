# Benchmark methodology

This document is written so that a skeptic can **reproduce or refute** the numbers in
[README.md](../README.md) and [FINDINGS.md](FINDINGS.md). If you run this and get a different
answer, the difference is the interesting part - please open an issue with your output.

Everything here describes a run performed on **2026-07-21**.

---

## 1. What was measured

One number, defined precisely:

```
eligible hit rate = sum(cache_read_tokens over rounds 2..N)
                    ---------------------------------------
                    sum(prompt_tokens     over rounds 2..N)
```

* **Token-weighted**, not the mean of per-round percentages.
* **Rounds 2..N only.** Round 1 is the cold cache write. See
  [Why the cold round is excluded](#5-why-the-cold-round-is-excluded).
* Both numerator and denominator come from the **upstream's own usage report**, relayed by the
  engine. Nothing in the harness estimates a token count.

Result: **99.53%** at a 748,918-token prefix over 4 rounds; **94.00%** at a 91,418-token prefix
over 5 rounds. Raw per-round records are in [FINDINGS.md](FINDINGS.md#finding-2-9953-token-weighted-prompt-cache-hit-rate-on-warm-rounds).

## 3. Prompt construction

The prompt has two parts: a large **stable prefix** that must be byte-identical every round, and
a small **growing tail** that simulates a conversation advancing.

**Prefix.** Generated by `build_prefix()` from nothing but an index:

```python
lines = ["# yangble5 cache-bench stable corpus (deterministic; do not summarize)"]
for i in range(target_tokens // 30):
    parity = "even" if i % 2 == 0 else "odd"
    lines.append(f"fact {i:06d}: the checksum of block {i} is {(i * 2654435761) % 10**9:09d} "
                 f"and its parity tag is {parity}.")
```

Three properties are load-bearing:

1. **Deterministic.** No uuid, no timestamp, no set/dict iteration order, no locale-dependent
   formatting. A single varying byte near the front of the prefix invalidates the entire cached
   prefix, which would make a perfectly good cache measure 0%.
2. **Poorly compressible, uniformly dense.** Numbered checksum lines keep tokens roughly
   proportional to lines. Natural-language filler repeats and compresses, and some upstreams
   normalise whitespace, which makes the token count drift unpredictably with size.
3. **Instructed not to be summarized.** The header exists because some upstream paths will
   happily "helpfully" compress a wall of facts.

**Tail.** Each round appends the previous exchange, so the prompt grows by a small constant. In
the 749K run the growth was **exactly 15 tokens per round** (748,918 -> 748,933 -> 748,948 ->
748,963). That growth is what the uncached remainder is made of, plus whatever fixed overhead the
upstream adds around the request.

**Session pinning.** Every round sends the same `metadata.user_id`
(`--session`, default `cache-bench-fixed-session`). This is the field the engine's
session-affinity logic reads, and it is what keeps all rounds on one account - which matters
because the upstream cache is **per account**. If you change `--session` between rounds you are
measuring a different thing (and will see repeated cold writes).

**Other request parameters:** `--max-tokens 48` (the completion is irrelevant; keep it small so
output tokens do not dominate cost), `--interval 2.0` seconds between rounds, `--timeout 300`.

### Why `--prefix-tokens` is a target, not a count

`build_prefix()` sizes the corpus at an assumed **30 tokens per line**. The live tokenizer
counted about **37**. So `--prefix-tokens 600000` produced a **748,918**-token prompt. Every
number the tool reports is the count the upstream returned; the 30-tokens-per-line figure is
only used to decide how many lines to emit. If you compare two providers, compare at equal
**measured** prompt sizes, not equal `--prefix-tokens`.

## 4. The denominator problem

This is the part most cache benchmarks get wrong, including ours before we looked.

Providers disagree about what `input_tokens` means, and the payload does not say which
convention you are being handed:

| Convention | `input_tokens` contains | Consequence |
|---|---|---|
| Gemini via CLIProxyAPI (verified live 2026-07-21) | `promptTokenCount`, which **already includes** the cached read | `input >= cache_read` |
| Anthropic native | only the **uncached** remainder | `cache_read` routinely exceeds `input` |

Divide `cache_read / input_tokens` blindly and you get either the right answer or a number above
100%, depending on who answered. The harness normalises:

```python
def prompt_denominator(input_tokens, cache_read_tokens):
    return input_tokens if input_tokens >= cache_read_tokens \
           else input_tokens + cache_read_tokens
```

Identical under both conventions, and incapable of reporting more than 100%. If you port this to
a third provider, check which convention it uses before trusting the output - and tell us, so we
can note it here.

## 5. Why the cold round is excluded

Round 1 of any session writes the cache. Its hit rate is 0% by definition, not by failure.

Including it makes the headline number **a function of how many rounds you ran**:

| Rounds run | Reported "hit rate" if the cold round is included |
|---:|---:|
| 4 (measured) | 74.6% |
| 10 (extrapolated) | ~89.6% |
| 40 (extrapolated) | ~97.0% |

Only the first row is a measurement. The other two are arithmetic on this run's per-round
figures, shown to make the point: none of them measured anything new about the cache. They
measured your choice of `--rounds`.
That is why the tool reports the warm figure as the headline, prints the cold round as its own
line, and gives you both numbers so you can compute whichever you want.

**When the cold round is the number you actually care about:** short-lived sessions. If your
workload starts a fresh conversation for every task, you pay a cold write every time and the
warm number is close to irrelevant to you. Prompt caching is a *long-session* optimisation, and
this benchmark measures long sessions on purpose.

The arithmetic for our run, so you can check the 74.6%:
`2,236,290 / 2,995,762 = 0.7465`, where 2,995,762 is the sum of all four rounds' prompt tokens.

## 6. Running it yourself

```bash
export YANGBLE5_API_KEY="the client key your engine accepts"   # env only; never a flag
export YANGBLE5_BASE_URL="http://127.0.0.1:8318"               # optional, this is the default

python tools/cache_bench.py --model yangble5 --prefix-tokens 600000 --rounds 4
```

The key is read from the environment and deliberately **not** accepted as a CLI flag: `argv` is
world-readable on most systems and lands in shell history.

Exit codes are designed so CI cannot confuse "bad score" with "broken run":

| Code | Meaning |
|---:|---|
| 0 | eligible hit rate >= `--target` (default 0.99) |
| 1 | below target, **or** no warm rounds were run |
| 2 | transport / HTTP / decode failure - the measurement never happened |

`--json` prints the full result object on stdout (progress moves to stderr), which is the form to
attach to an issue.

### Running it against a different provider

The harness only needs an Anthropic-format `/v1/messages` endpoint that reports usage:

```bash
# any OpenAI/Anthropic-compatible gateway that speaks /v1/messages
export YANGBLE5_BASE_URL="http://127.0.0.1:9999"
export YANGBLE5_API_KEY="that gateway's key"
python tools/cache_bench.py --model <their-model-name> --prefix-tokens 600000 --rounds 4 --json
```

For a comparison to mean anything, hold these equal across providers:

* **measured** prompt size (adjust `--prefix-tokens` per provider until the reported
  `prompt_total` matches, since tokenizers differ),
* `--rounds`, `--max-tokens`, `--interval`,
* one session id for the whole run,
* and report the cold round separately for both.

Then state which convention each provider used for `input_tokens` (section 4). A provider that
reports no cache fields at all is not scoring 0% - it is unmeasurable on this path, and the tool
says so rather than printing a zero.

## 7. Known confounds

Every one of these can move the number, and none of them were controlled for:

* **Provider-side load.** Shared infrastructure, no isolation, no repetitions. This is the
  biggest single source of variance in the latency column, and it is why we call the latency
  figures an anecdote.
* **Implicit cache TTL.** We do not know the upstream's cache lifetime. `--interval 2.0` keeps
  rounds close together; a long pause between rounds may expire the entry and score a cold
  write mid-run.
* **Account switching.** If session affinity is off, or the TTL expires, or a credential cools
  down mid-run, a later round can land on a different account with a cold cache. The run looks
  broken rather than subtly wrong, which is the good outcome, but check `--session` and your
  `routing` block before blaming the upstream.
* **Engine restart.** The session-to-credential table is in memory. Restarting the engine
  mid-run re-binds sessions.
* **Cache granularity.** The uncached remainder was ~3.5K tokens at a 749K prefix and ~5.4K at a
  91K prefix. That is upstream behaviour, not ours, and it is why the ratio is prefix-size
  dependent.
* **Single run, single machine, single afternoon.** No repetitions, no confidence intervals.
  Treat every figure as one sample.
* **This costs real money.** The 4-round 749K run moved **2,995,762 prompt tokens** through a
  billed upstream (of which 2,236,290 were cache reads, which are cheaper but rarely free).
  Check your provider's cached-input price before running it repeatedly, and start with the
  default `--prefix-tokens 30000` to confirm the plumbing works.

## 8. How to refute us

Concrete, falsifiable claims and what would break each one:

| Claim | Refuted by |
|---|---|
| Warm token-weighted hit rate reaches ~99.5% at a ~749K prefix on this upstream | A run at an equivalent measured prompt size, same session, that reports materially lower with cache fields present |
| The uncached remainder is roughly constant rather than proportional | A run where the remainder scales with prefix size |
| `nextModelPoolOffset` selects pool members without consulting routing or session | Reading `conductor.go` in 7.1.23 and finding a session identifier in the key |
| A mid-conversation `role: "system"` 400s streaming but not non-streaming on 7.1.23 | The `stream: true` / `stream: false` replay in [FINDINGS.md](FINDINGS.md#how-to-reproduce-and-bisect-it-yourself) behaving identically |
| 748,918 tokens were ingested without truncation | An upstream `input_tokens` materially below what was sent |

What would **not** refute anything: a different absolute hit rate at a different prefix size, on
a different provider, or with a different session policy. Those are different measurements. Say
which one you ran.

## 一、PTT

### 標題

```
[心得] 我的 LLM proxy 把同一段對話的快取切成兩半
```

備選(想讓標題更明確一點):

```
[心得] 追一個 LLM proxy 快取失效的 bug,結果在原始碼裡
```

### 內文

先講結論,趕時間的可以只看這段:

如果你在自架的 LLM proxy 裡,把**一個模型別名對應到兩個上游模型名稱**(pool),
那你同一段對話的 prompt cache 很可能正在被切成兩半,而且**不會噴任何錯誤**。

---

**怎麼發現的**

我平常拿 Claude Code 跑很長的 session,某天開始覺得不太對:
prompt 幾乎跟上一輪一模一樣,但回報的 cached token 數趨近於零,
而且大概一半的 request 直接 502。

一開始當然先懷疑自己設定寫錯,結果翻了半天設定檔都是對的,最後只好去看引擎的原始碼。

**原因在原始碼裡**

用的引擎是 CLIProxyAPI(MIT 的 Go 專案,**不是我寫的**)。在 7.1.23 版裡,
當一個別名對應到 `openai-compatibility` pool 裡的兩個上游模型名稱時,
每次 request 要打去哪個上游,是由一個**全域輪替計數器**決定的:

```
sdk/cliproxy/auth/conductor.go
  nextModelPoolOffset(...)        # 遞增後回傳的計數器
  modelPoolOffsets                # 計數器狀態
  openAICompatModelPoolKey(...)   # key 是「pool」,不是 session
```

重點是那個 key 是**以 pool 為單位**,裡面**完全沒有 session id、對話 id 或憑證 id**。
所以會有兩個直接後果:

* `routing.strategy` 在選 pool 成員這件事上**根本沒被讀到**
* `session-affinity` 也沒有——它綁的是「session 對應到哪個帳號」,
  **從來就不是「session 對應到 pool 裡哪個成員」**

我兩個都設了,兩個都沒作用。

**為什麼這會毀掉快取**

關鍵事實:**上游的 prompt cache 是「按模型、按帳號」隔離的。**
快取不在 proxy 裡,也不在這個 repo 裡,proxy 能做的只有「確保同一段對話的連續請求打到同一個地方」。

所以一段對話被拆到兩個上游 = 快取被切成兩半。而且是永久的:
第 k 次請求打到 A,它最多只能讀到第 k-2 次請求寫進 A 的內容,
中間那一輪新增的東西(最後一則使用者訊息、最後一則回覆、最後一個工具結果)全部不在 A 裡,在 B 裡。

**最可怕的是它不會報錯。** 我這次會發現純粹是運氣:
我 pool 的第二個成員 `gemini-3-pro-high` 在那個 provider 上根本沒註冊,
所以每次輪到它就 502,一半的請求直接失敗,我才注意到。
如果 pool 裡兩個成員**都是合法的**,它會**安安靜靜地讓你付兩倍的錢**,你完全不會發現。

**自己檢查(不用裝 Go)**

```bash
strings cli-proxy-api.exe | grep -E 'nextModelPoolOffset|modelPoolOffsets|conductor\.go'
```

有東西出來就代表這段程式碼在你正在跑的那顆執行檔裡。然後請去看**你自己那版**的原始碼,
我只讀過 7.1.23,別的版本行為可能不一樣。

**修法**

改成 provider channel 上的 1:1 直連別名(`oauth-model-alias`),不要用 pool。另外三個設定要改:

```yaml
routing:
  strategy: "fill-first"        # 出廠預設是 round-robin
  session-affinity: true        # 出廠預設是 false
  session-affinity-ttl: "12h"   # 出廠預設是 1h
```

順帶一提,**這三個出廠預設值對長時間的 agent session 都是不利的**,
不管你用哪套 proxy 都值得回頭看一下自己的設定。

選 `fill-first` 的理由:session 對帳號的對應表是存在記憶體裡的,引擎一重啟就空了。
`fill-first` 會固定重新挑到第一個健康的憑證,通常還是同一個帳號,上游快取還是熱的;
`round-robin` 則會把重啟後的流量打散到不同帳號,每個都付一次冷寫入。
(這段是從兩種策略的語意推論的,**我沒有實測重啟行為。**)

**修好之後實測到的數字**

單機、單次、2026-07-21,用 repo 裡附的腳本量的。逐輪原始數字,沒有平均也沒有修飾:

| 輪次 | prompt tokens | 快取讀取 | 命中率 | 延遲 |
|---:|---:|---:|---:|---:|
| 1(冷) | 748,918 | 0 | 0.00% | 21,410 ms |
| 2 | 748,933 | 745,438 | 99.53% | 10,753 ms |
| 3 | 748,948 | 745,430 | 99.53% | 23,457 ms |
| 4 | 748,963 | 745,422 | 99.53% | 22,381 ms |

**暖輪次 token 加權命中率 99.53%**,而且那個 **748,918 token 的 prompt 沒有被截斷**。

**以下是會被電的地方,我自己先講**

* **99.53% 只涵蓋暖輪次(2-4 輪)。** 第一輪是冷寫入,0% 是理所當然的,
  而且**每個 session 都一定會付一次**。把冷輪次算進去這次是 74.6%,
  但那個數字其實只反映「我跑了幾輪」——跑 40 輪會變成大約 97%,卻沒有多量到任何東西。
* **99.53% 是「上限」,不是常態值。** 量測工具模擬的對話每輪只長 **15 個 token**,
  這是對快取最有利的形狀。真實 agent 每一輪會塞進工具輸出、檔案內容或 diff,
  多出來的都是沒快取到的量,比例一定更低。
* **命中率跟 prefix 大小有關。** 沒被快取的殘量大約是固定的(749K prefix 時約 3.5K token),
  不是等比例的,所以 prefix 越大比例越好看。這個「方向」有觀察到,
  但**其他 prefix 大小的實際數字不在已公開的證據裡**——公開的就只有 748,918 這一次跑,
  所以我不會報別的數字。工具預設的 30K 根本到不了 99%。
  **所以 99.53% 不能當成通用數字引用。**
* **「pool 會把你的命中率壓在 5 成左右」這句是我從原始碼推出來的結構上限,不是實測。**
  我**沒有**做 pool vs. 直連的對照實驗。原因是我搞懂機制的時候,那個 pool 設定已經因為
  另一個不相關的原因壞掉了(第二個成員根本不存在),再蓋一個已知壞掉的設定去跑 benchmark
  要燒掉好幾百萬 token,不值得。**我實測的只有「修好之後」。**
* **單機、單次、一個下午。** 沒有重複實驗、沒有信賴區間、沒有跨供應商對照。
* **延遲沒有變好。** 第 2 輪比冷啟動快了大約 2 倍,但**第 3、4 輪反而比冷啟動還慢**,
  而它們可是讀了 99.5% 的快取。快取省的是**錢**,省時間這件事在這次的證據上並不成立。
* **完全沒有即時網搜。** 實測 2026-07-21:問「現在是哪一年」,
  Gemini 上游答 **2024**、Grok 上游答 **2025**。所有回答都是訓練截止前的參數記憶。
  需要即時資訊請用別的環境,不要用這套。
* **底層那顆 Go 引擎 CLIProxyAPI 不是我寫的**,MIT 授權,做掉了所有協定層的苦工。
  覺得這篇有用的話請**先去 star 那個專案**,沒有它這裡什麼都不是。
  我寫的是:那個發現、設定、一個 shim、兩支量測工具。

**順便,兩個 Claude Code 的雷**

1. **Claude Code 對它不認識的模型名稱一律當成 200K**,會提早自動壓縮,
   而**每次壓縮都是一次完整的 prompt 重寫,順便把快取毀掉**。
   要設 `CLAUDE_CODE_MAX_CONTEXT_TOKENS=1000000`(官方環境變數,v2.1.193+)。
   但要講清楚:**把數字設大不會憑空變出上下文**,只是改變客戶端決定壓縮的位置,
   上游吃不下還是會截斷或報錯。我驗證到 748,918,**沒有驗證 1,000,000**。

2. **`400 Request contains an invalid argument` 這個錯不是「偶發」的。**
   Claude Code 2.1.x 之後會在 messages 陣列**中間**塞一則 `role: "system"` 的訊息,
   而 7.1.23 的 antigravity **串流**翻譯器是原封不動把 role 傳下去的,
   Gemini 的 `streamGenerateContent` 直接拒絕。**非串流路徑則可以接受同樣的東西。**
   所以它跟著**傳輸方式**跑,不是跟著內容跑——難怪照內容二分法怎麼切都切不出來。

   > 通用的除錯手法(這才是重點):把失敗的 request body 抓下來,用 curl 重送兩次,
   > **只改 `"stream": true` 成 `false`**。如果非串流過、串流 400,而內容是 byte 相同的,
   > 那就**不要再切你的 prompt 了**,問題在串流翻譯路徑上。
   > 我就是先切內容才浪費掉一個下午的。

   上游在 **v7.2.93** 修好了(把 `system` 對應成 `user`)。舊版才需要 repo 裡那個 shim,
   **升上去就把 shim 刪掉。**

### Threads 注意事項

* **不要**在任何一則裡出現「免費」「無限」「破解」「白嫖」,也不要提任何具體的免費額度金額。
* **不要**寫成「台灣自己訓練的模型」或任何暗示自有模型的說法。這是**閘道**,不是模型。
* 有人問「所以是免費的 Claude 嗎」→ 回:不是,token 一樣是算在你自己設定的上游帳號上,
  這只是讓你別把已經付過的錢浪費掉。
* 有人問「這樣不會被 ban 嗎」→ **不要迴避**。回:如果你拿個人免費帳號去撐公開服務,
  會,而且被 ban 的是**帳號**不是服務,連你那個帳號的其他東西一起沒。
  repo 裡有一份專門講這件事的文件,建議的做法是 BYOK 或有正式授權的付費金鑰。
* 圖片建議用那張逐輪數字表,**不要**做成只有「99.53%」四個大字的圖卡。
  單獨一個百分比脫離 prefix 大小和暖輪次的脈絡就是誤導,而且它會被轉走。
